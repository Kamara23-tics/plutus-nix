1. üì¶ Imports Overview

Plutus API Modules

  • Plutus.V2.Ledger.Api: Provides fundamental types including POSIXTime, PubKeyHash, BuiltinByteString, transaction context (ScriptContext, TxInfo), and validator construction types.
  • Plutus.V2.Ledger.Contexts: Contains utility functions for transaction context validation including txSignedBy, getContinuingOutputs, findOwnInput, valuePaidTo, and txOutDatum.
  • Plutus.V1.Ledger.Interval: Supplies interval checking functions (contains, from) for time-based validation of milestone deadlines and campaign timing.
  • Plutus.V1.Ledger.Value: Provides valueOf for extracting specific token amounts, plus adaSymbol and adaToken for ADA calculations in donations and disbursements.

PlutusTx Modules

  • PlutusTx: Core module enabling on-chain compilation with Template Haskell (compile, unstableMakeIsData, unsafeFromBuiltinData) for all data types.
  • PlutusTx.Prelude: Plutus-compatible prelude with safe on-chain functions including list operations (foldr, length, !!), comparison operators, and control flow.
  • PlutusTx.Builtins: Low-level builtin functions for on-chain operations, particularly toBuiltin for converting between Haskell and Plutus types.

Serialization and Encoding

  • Codec.Serialise: CBOR serialization for compiling validators to binary format suitable for blockchain deployment.
  • Data.ByteString modules (Lazy, Short, Strict): Handle binary data in various formats for efficient validator serialization, hashing operations, and report verification.

Cardano API

  • Cardano.Api: Provides network configuration (NetworkId, NetworkMagic), script hashing (hashScript), and address creation functions for testnet/mainnet deployment.
  • Cardano.Api.Shelley: Contains Shelley-era specific types like PlutusScriptSerialised and address construction functions (makeShelleyAddressInEra) for Babbage-era addresses.

Standard Haskell Libraries

  • Prelude: Basic I/O operations (IO, putStrLn), string manipulation, and file operations, imported qualified to avoid conflicts with PlutusTx.Prelude.
  • Data.Text: Efficient text handling for address conversion and serialization operations.

2. üóÉÔ∏è Data Structures

Milestone

Represents a single milestone within a grant program:

  • mAmount: Payment amount for this milestone in lovelace (1 ADA = 1,000,000 lovelace)
  • mReportHash: Required report hash that must be submitted before funds release
  • mDeadline: POSIX timestamp deadline for milestone completion
  • mCompleted: Boolean flag indicating whether the milestone has been completed

Grant

Represents a grant program with milestone-based disbursement:

  • gRecipient: Public key hash of the grant recipient who receives milestone payments
  • gTotalAmount: Total grant amount in lovelace (sum of all milestone amounts)
  • gMilestones: List of milestones defining the disbursement schedule
  • gCurrentPhase: Current milestone index (0-based) tracking progress
  • gTotalReleased: Cumulative amount released so far across all completed milestones
  • gDescription: Grant description or reference identifier

Campaign

Campaign treasury configuration with spending controls:

  • cTreasurer: Public key hash of the campaign treasurer authorized to manage funds
  • cPerTxLimit: Maximum spending allowed per single transaction in lovelace
  • cTotalLimit: Total campaign spending limit/budget in lovelace
  • cDisclosureRef: IPFS or Arweave reference to campaign disclosure documents
  • cTotalSpent: Running total of all campaign spending for budget tracking
  • cTotalDonations: Running total of all donations received
  • cActive: Boolean flag indicating whether the campaign is currently active

DonorRecord

Immutable record of a single donation for transparency:

  • drDonor: Public key hash of the donor
  • drAmount: Donation amount in lovelace
  • drTimestamp: POSIX timestamp when donation was made

CampaignDatum

Combined datum containing all campaign state:

  • cdCampaign: The Campaign configuration and current state
  • cdGrants: List of all Grant programs under this campaign
  • cdDonors: Immutable list of DonorRecord entries forming an audit trail

CampaignAction

Redeemer actions defining all possible contract interactions:

  • Donate: Anyone can donate to the campaign (includes timestamp)
  • DisburseOperational: Treasurer disburses funds for operational expenses (includes amount)
  • CreateGrant: Treasurer creates a new grant program (includes Grant details)
  • ReleaseMilestone: Release milestone payment (includes grant index and report hash)
  • UpdateDisclosure: Update campaign disclosure document reference
  • CloseCampaign: Treasurer closes the campaign (no further operations allowed)

3. üß† Core Validator Logic

Helper Functions

  • findGrant: Safely retrieves a grant by index from the grant list
  • updateGrantAt: Updates a specific grant at a given index in the list
  • getCurrentMilestone: Gets the current active milestone for a grant
  • updateMilestoneAt: Marks a specific milestone as completed
  • totalDonations: Calculates sum of all donations from donor records
  • milestonePastDeadline: Checks if a milestone has exceeded its deadline

mkValidator

The main validator function implements six distinct transaction flows:

1. Donate (Donate timestamp)

Purpose: Allow anyone to contribute funds to the campaign.

Validation Conditions:

  • Campaign must be active (not closed)
  • Donation must actually increase the UTXO value at script address
  • New donor record must be added to the immutable audit trail
  • Campaign's total donations counter must be incremented correctly

2. Disburse Operational (DisburseOperational amount)

Purpose: Allow treasurer to spend funds for campaign operations.

Validation Conditions:

  • Campaign must be active
  • Transaction must be signed by the campaign treasurer
  • Amount must not exceed per-transaction spending limit
  • Amount must not cause total spending to exceed campaign budget
  • Treasurer must receive the specified amount in the transaction
  • Campaign's total spent counter must be updated with the new amount

3. Create Grant (CreateGrant grant)

Purpose: Allow treasurer to establish a new milestone-based grant.

Validation Conditions:

  • Campaign must be active
  • Transaction must be signed by the campaign treasurer
  • Grant total amount must fit within remaining campaign budget
  • Grant must be added to the campaign's grant list
  • Milestone structure must be valid (sum of milestones equals grant total, at least one milestone)

4. Release Milestone (ReleaseMilestone grantIdx reportHash)

Purpose: Release milestone payment after report submission.

Validation Conditions:

  • Campaign must be active
  • Transaction must be signed by the campaign treasurer
  • Grant index must be valid (within bounds of grant list)
  • Submitted report hash must match the milestone's required report hash
  • Milestone must not already be marked as completed
  • Current time must be before the milestone deadline
  • Grant recipient must receive the milestone amount
  • Milestone must be marked as completed in continuing output
  • Grant's total released and current phase must be updated correctly

5. Update Disclosure (UpdateDisclosure newRef)

Purpose: Allow treasurer to update campaign disclosure documents.

Validation Conditions:

  • Transaction must be signed by the campaign treasurer
  • Campaign's disclosure reference must be updated in continuing output

6. Close Campaign (CloseCampaign)

Purpose: Treasurer closes campaign, preventing further operations.

Validation Conditions:

  • Transaction must be signed by the campaign treasurer
  • Campaign must be marked as inactive (cActive = False) in continuing output
üîí Transparency Note: All donations are permanently recorded in an immutable audit trail. The donor list is append-only, ensuring complete transparency and preventing retroactive modifications of the funding record.

4. ‚öôÔ∏è Validator Script Compilation

mkValidatorUntyped

Wraps the typed validator function for compatibility with Plutus on-chain execution using BuiltinData. This untyped wrapper:

  • Deserializes datum (CampaignDatum) from raw builtin data
  • Deserializes redeemer (CampaignAction) from raw builtin data
  • Deserializes script context from raw builtin data
  • Calls the typed mkValidator function with deserialized parameters
  • Returns () on successful validation or triggers an error on failure

validator

Compiles the untyped validator using Template Haskell's compile splice into a Plutus Core script ready for blockchain deployment. Uses mkValidatorScript to wrap the compiled code into a proper Validator type.

5. üîß Helper Functions

Script Hash and Address Generation

  • plutusValidatorHash: Computes the on-chain validator hash. Serializes the validator, converts through ShortByteString ‚Üí ByteString ‚Üí BuiltinByteString, and wraps in ValidatorHash.
  • plutusScriptAddress: Generates the Plutus script address with ScriptCredential and no stake address, used for on-chain address representation.

Bech32 Address Generation

  • toBech32ScriptAddress: Generates the validator script's human-readable Bech32 address for CLI and wallet integration. Serializes validator, creates PlutusScriptV2, hashes it, and constructs a Babbage-era address.

File Operations

  • writeValidator: Writes the compiled validator script to campaign-finance.plutus file in CBOR format for deployment.

Main Function

  • main: Orchestrates validator compilation, file writing, and information display. Sets up testnet configuration, generates addresses, and prints comprehensive feature documentation.

7. üß∑ Testing Strategy

Donation Testing

  • Test donations with various amounts
  • Verify donor records are appended correctly
  • Test donation total updates accurately
  • Verify donations to inactive campaigns fail
  • Test multiple donations from same donor
  • Verify UTXO value increases correctly

Operational Disbursement Testing

  • Test disbursements within per-tx limit
  • Test disbursements exceeding per-tx limit (should fail)
  • Test disbursements within total budget
  • Test disbursements exceeding total budget (should fail)
  • Verify treasurer signature requirement
  • Test spending counter updates correctly
  • Verify unauthorized disbursement attempts fail

Grant Creation Testing

  • Test grant creation with valid milestone structure
  • Test grants with milestone sums not matching total (should fail)
  • Test grants with zero milestones (should fail)
  • Test grants exceeding remaining budget (should fail)
  • Verify grant is added to list correctly
  • Test multiple grants under one campaign

Milestone Release Testing

  • Test milestone release with correct report hash
  • Test milestone release with wrong report hash (should fail)
  • Test release before deadline (should succeed)
  • Test release after deadline (should fail)
  • Test releasing already completed milestone (should fail)
  • Test invalid grant index (should fail)
  • Verify recipient receives correct amount
  • Verify phase advancement and total released updates
  • Test sequential milestone releases

Campaign Lifecycle Testing

  • Test disclosure updates by treasurer
  • Test unauthorized disclosure updates (should fail)
  • Test campaign closure
  • Test operations on closed campaign (should fail)
  • Verify complete campaign lifecycle from creation to closure

Edge Cases

  • Test with maximum per-tx limit
  • Test with many grants (list handling)
  • Test with long donor lists (gas considerations)
  • Test milestone at exact deadline boundary
  • Test with zero amount operations
  • Test concurrent transactions on same UTXO

8. ‚úÖ Best Practices

Smart Contract Development

  • Use descriptive traceIfFalse messages for debugging
  • Implement comprehensive authorization checks for all privileged actions
  • Use continuing outputs to maintain state across transactions
  • Validate all numeric operations to prevent overflow/underflow
  • Test boundary conditions extensively (limits, deadlines, indices)
  • Consider gas costs when managing large lists (donors, grants)

Security Considerations

  • Always verify treasurer signature for privileged operations
  • Enforce spending limits at both per-transaction and total levels
  • Use immutable donor records to prevent audit trail tampering
  • Validate grant and milestone structures before creation
  • Implement proper deadline checking to prevent late releases
  • Verify report hashes to ensure milestone completion requirements
  • Prevent operations on closed campaigns
  • Use cryptographic hashes for report verification

Transparency Best Practices

  • Maintain complete, append-only donor records
  • Store disclosure documents in permanent storage (IPFS/Arweave)
  • Provide real-time spending and donation dashboards
  • Display milestone progress publicly
  • Enable anyone to verify report hashes
  • Publish regular campaign updates via disclosure references
  • Allow easy auditing of all transactions

User Experience

  • Build intuitive donation interface with clear receipt generation
  • Display current spending vs budget in real-time
  • Show milestone progress bars for each grant
  • Provide donor breakdown and statistics
  • Enable easy access to disclosure documents
  • Implement notification system for milestone releases
  • Create public audit trail viewer
  • Display remaining budget and per-tx limits clearly

Integration Tips

  • Use reference inputs for read-only campaign data access
  • Cache frequently accessed data to reduce blockchain queries
  • Implement IPFS/Arweave integration for disclosure documents
  • Build APIs for donor and spending analytics
  • Create webhook notifications for campaign events
  • Enable CSV export for donor records and spending history
  • Implement report hash generation tools
  • Provide treasurer dashboard for campaign management
